# /case-guide - 定型案件の情報取得ガイド

企業名を指定 → スプシ+Slack情報を収集 → フェーズ判定 + 詳細チェックリスト生成。
case-alertのフェーズ遷移通知で概要を受け取った後、詳細が必要なときに手動実行する。
Eisukeが案件管理DBで「ガイド要求」にチェック → pollモードで自動生成も可能。

## Input

`$ARGUMENTS` を解釈する:
- `poll` → 案件管理DBの「ガイド要求=true」のエントリを処理
- `<企業名>` → 企業名を指定して実行
- 空 or `help` → 使い方を表示して終了

## 設定

設定ファイルを読み取る: `~/.claude/private/case-config.md`

- **案件管理DB data source ID**: case-config.md参照
- **回収リスト**: case-config.md参照
- **GOG_ACCOUNT**: case-config.md「実行ユーザー」セクション参照
- **フローチャート v0.2**: case-config.md参照

---

## 処理フロー

### Step 0: Poll モード（`$ARGUMENTS` = `poll` の場合）

案件管理DBから「ガイド要求=true」のエントリを取得:

```
mcp__notion__notion-query-data-sources({
  data_source_id: "<case-config.mdのDB data source ID>",
  query: "SELECT * FROM \"collection://<case-config.mdのDB data source ID>\" WHERE ガイド要求 = true LIMIT 1"
})
```
→ エントリがなければ「未処理案件なし」で終了
→ エントリがあれば:
  - `page_id` を記録（完了時の更新用）
  - `企業名` を取得 → 以降のStep で使用
  - DB上の DPD, 債権額, フェーズ, 担当者 等を初期値として取得

### Step 1: 案件管理DBから企業情報を取得（直接実行モード）

直接実行モード（`$ARGUMENTS` = `<企業名>`）の場合のみ:

```
mcp__notion__notion-query-data-sources({
  data_source_id: "<case-config.mdのDB data source ID>",
  query: "SELECT * FROM \"collection://<case-config.mdのDB data source ID>\" WHERE 企業名 LIKE '%<企業名>%' LIMIT 5"
})
```
→ DB登録済みなら DPD, 債権額, フェーズ, 担当者, MT残高🤖 等を取得
→ 未登録ならStep 2でスプシから直接取得

### Step 2: 回収リストから最新情報を取得

```bash
GOG_ACCOUNT=<case-config.mdのGOG_ACCOUNT> gog sheets metadata <case-config.mdの回収リストID> --json
```
→ 回収状況シート名を特定し、企業名で全シート検索（case-config.md列マッピング参照）
→ DPD計算: `今日 - 支払期日`
→ DBデータがあればスプシの最新値で補完

### Step 2.5: MT残高取得

Step 1 or 2 で orgID が判明している場合、Metabase から MT残高を取得する。

```bash
bash "$HOME/.claude/scripts/metabase-mt-balance.sh"
```

→ 成功時: `~/.claude/private/tmp_mt_balance.json` から対象orgIDの残高を取得
→ 失効/エラー時: MT残高=不明として処理続行（行動推奨はフェーズ×債権額のみで判定）

保全実効性を自動判定（case-config.md「保全実効性の自動判定」参照）:
- MT残高 >= 債権額の50% → 高
- MT残高 >= 100万 → 中
- MT残高 < 100万 → 低
- MT残高 = 不明 → 不明

### Step 4: Slack検索（補足情報の収集）

```
mcp__slack__slack_search_messages({ query: "<企業名>" })
```
→ 直近のSlackやりとりから追加情報を収集:
- 先方との連絡状況
- GP対応期間の記録
- 既に取得済みの資料
- これまでの経緯

### Step 5: フェーズ判定

フローチャート v0.2 のロジックに基づき判定:

```
DPD 0-4:
  → フェーズ0（GP対応中）
  → GPが初回架電・メール対応中

DPD 5-7:
  → 債権額 1,000万以上? → 非定型案件 → Taroエスカレーション
  → 債権額 1,000万未満 → フェーズ1（Credit引継ぎ・初期情報収集）

DPD 5-12:
  → 先方と接触済み? → フェーズ2（接触・主張確認）
  → 未接触? → フェーズ1 継続

DPD 7-19:
  → 主張確認済み? → フェーズ3（裏付け確認・支払計画）
  → 未確認? → フェーズ2 に戻る

支払予定日設定済み:
  → フェーズ4（履行監視）

エスカレーション条件該当:
  → フェーズ5（エスカレーション判定）

エスカレーション条件:
  1. 先方が弁護士を立てた
  2. 債務を否認
  3. 連絡不通10営業日以上
  4. 分割2回目不履行
  5. DPD30経過で未解決
  6. 民事再生・破産の言及
  7. 債権額が事後的に1,000万以上と判明
  8. 裏付け資料の意図的提出拒否
```

### Step 5.5: 行動推奨生成

case-config.md の「行動推奨マトリクス」を参照し、以下を生成:

1. **債権額レベル判定**（S/A/B/C）:
   - S: >= 3,000万 / A: >= 1,000万 / B: >= 300万 / C: < 300万
2. **フェーズ × 債権額レベル → 基本推奨アクション**: マトリクスから取得
3. **MT残高による上書き判定**（Step 2.5 でMT残高が取得できた場合のみ）:
   - MT残高 >= 債権額の50% かつ フェーズ3+ → ⚠️「保全機会: 仮差押を即検討」を追加
   - MT残高 >= 100万 かつ 保全ステータス=なし かつ DPD>=13 → ⚠️「保全機会あり」を追加
   - MT残高 < 10万 → 「MT残高僅少: 保全効果限定的」を注記
4. **急げ度判定**:
   - 🔴即日: 債権額レベルS or MT残高>=債権額50%で保全未済
   - 🟡今週: 債権額レベルA or DPD30+
   - 通常: 債権額レベルB/C

出力値（Step 7・8 で使用）:
- `急げ度`: 🔴即日 / 🟡今週 / 通常
- `推奨アクション`: マトリクスから取得した推奨アクション文
- `判定根拠`: フェーズ・債権額・MT残高に基づく根拠
- `保全推奨`: MT残高が取れた場合のみ（なければ空）

### Step 6: チェックリスト生成

フェーズ + 先方主張に応じたチェックリストを生成:

#### フェーズ0 チェックリスト
- [ ] 延滞Masterで債務者名・金額・延滞日数・過去支払履歴を確認
- [ ] 回収リストへのGPの記入内容をモニタリング
- [ ] 契約書の支払条件（支払期日・遅延損害金条項）を確認
- [ ] DPD5到達時: 500万以上 or 連絡不通 → Credit引継ぎ

#### フェーズ1 チェックリスト
- [ ] GP対応期間の情報を引き継ぎ（架電メモ・メール履歴・回収リスト記入内容）
- [ ] 過去の延滞履歴の確認
- [ ] 当社側担当者（CS・営業）の特定
- [ ] **DPD5+チェックポイント**: 債権額1,000万以上? → YES → Taroにエスカレーション
- [ ] MT連携で口座動態の確認（直近の入出金パターン）
- [ ] Credit架電の実施

#### フェーズ2 チェックリスト（共通）
- [ ] 先方の主張を具体的に確認（「いつまでに払えるか」「なぜ遅れているか」）
- [ ] 主張内容を記録（通話メモ / メール / 回収リスト）
- [ ] **債権額500万以上**: 接触段階で税務申告書（直近期）を要求
- [ ] 連絡不通の場合: 登記情報で本店所在地・代表者確認 → 書面催告

##### 主張パターン別チェック

**「○日までに払う」（具体日あり）の場合:**
- [ ] その資金の原資は何かを確認
- [ ] （500万以上）税務申告書を要求

**「取引先からの入金待ち」の場合:**
- [ ] どの取引先か、入金予定日はいつかを確認
- [ ] 当該取引の契約書・発注書を要求
- [ ] 税務申告書を要求

**「資金調達中」の場合:**
- [ ] 調達手段（銀行融資/VC増資/資産売却/その他）を確認
- [ ] 完了見込み時期を確認
- [ ] 金融機関の審査状況を要求
- [ ] 税務申告書を要求

**「和解金/保険金で払う」の場合:**
- [ ] 相手方/保険会社名、金額、支払時期を確認
- [ ] 訴訟概要、和解書/保険請求書を要求

**「分割で払いたい」の場合:**
- [ ] 月額いくら、何回、初回支払日を確認
- [ ] 直近3ヶ月の月次損益を要求
- [ ] 税務申告書を要求

**「弁護士に相談中」の場合:**
- [ ] 弁護士の事務所名・連絡先を確認
- [ ] → **即Taroにエスカレーション**

**「払う義務はない」（債務否認）の場合:**
- [ ] 具体的な理由を確認
- [ ] → **即Taroにエスカレーション**

#### フェーズ3 チェックリスト
- [ ] 主張の裏付け資料を要求（提出期限: 5営業日）
- [ ] **全パターン共通（500万以上）**: 税務申告書（直近期）の要求
  - 確認: 長期/短期借入金の増減、借入先金融機関、銀行口座情報
- [ ] 資金繰り表（直近3ヶ月）の要求
- [ ] 資料が出てきたら内容確認（支払予定日が現実的か、金額が合っているか）
- [ ] 借入状況・口座情報から保全措置の可否を判断
- [ ] 支払計画の合意:
  - 一括: 支払期日をメールで確認
  - 分割: 月額・回数・初回日（最長6ヶ月原則、超過はTaro相談）
  - 月額が売上の20%以上 → Taro相談
  - 初回支払日が1ヶ月以上先 → 要注意

##### 資料不提出時の対応
- 連絡不通 → フェーズ5へ
- 「出せない」（能力不足）→ フォーマット提供・口頭ヒアリングで代替 → 3営業日後再確認
- 「出したくない」（意図的拒否）→ Taroに相談（仮差押示唆を検討）→ フェーズ5へ

#### フェーズ4 チェックリスト
- [ ] 支払予定日の2営業日前にリマインドメール送付
- [ ] 支払予定日の翌営業日に入金確認
- [ ] 分割の場合: 毎月の支払日翌営業日に入金確認
  - 2回連続期日通り → リマインド省略OK
  - 1回でも遅延 → 即フォロー架電

##### 不履行時
- 翌営業日に架電
  - 「3営業日以内に払う」→ 1回のみ猶予
  - 具体日なし → フェーズ5へ
  - 新たな主張 → フェーズ2に戻る（2周目はTaro報告）
- 分割2回目不履行 → 即フェーズ5

#### フェーズ5 チェックリスト
- [ ] エスカレーション報告書の作成（フォーマット準拠）
- [ ] Taroへの即報告
- [ ] 法的措置の財産基準チェック:
  - 債権額100万以上 かつ 口座残高100万以上 → 法的措置（仮差押等）検討
  - 上記未満 → 内容証明送付で割り切り

### Step 7: Notionページ作成

`mcp__notion__notion-create-pages` で結果ページを作成:

- 親ページ: case-config.md の DB parent page
- タイトル: `案件調査ガイド_<企業名>_YYYYMMDD`
- フォーマット:

```

  **企業名**: <企業名> / **債権額**: ¥XXM / **DPD**: XX
  **フェーズ**: <フェーズ番号と名称>

  <推奨アクション>
  理由: <判定根拠>
</callout>

<MT残高が取得できた場合のみ以下を出力>
<callout icon="🏦" color="yellow_bg">
  MT残高: ¥XXM / 保全実効性: 高/中/低/不明
  <保全推奨がある場合> ⚠️ <保全推奨内容>
</callout>
```

## Notes
- AskUserQuestion は使用しない（非対話で完結）
- エラー時も可能な範囲で続行。取得できたデータのみでガイドを生成
- フェーズ判定はDPDを主要指標とするが、Slack情報から実際の進捗を加味して調整
- 1,000万以上の案件は非定型として即Taroエスカレーション（ガイド生成は行うが冒頭に警告）
- 金額はMM表記（例: 15.2M）
- 企業名の部分一致で検索（スプシ・Slack両方）
