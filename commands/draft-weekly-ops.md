# 本部長会議 週次ページ自動下書き生成

引数: `$ARGUMENTS`（オプション: 対象日付 `YYYYMMDD`。未指定なら次の水曜日）

## 重要ルール
- AskUserQuestion は使用しない（非対話で完結）
- すべてのステップを自動実行する
- エラーが発生しても可能な範囲で続行する
- データ不足の本部は「（データ不足: チャネル未マッピング or メッセージなし）」と明記
- 天気図の判定に自信がない場合は ☁ をデフォルトとする
- **対象期間が3日未満の場合**: データ充足本部（チャネル投稿5件以上）を除き、天気図は一律 ☁（データ不足）とする
- **表記ルール**: Notionページ内では「支払い事業本部」と表記すること（「支払い.com」はURL自動リンクされるため禁止）
- **意思決定志向**: このページの目的は「情報共有」ではなく「本部長が意思決定する場を作る」こと。すべてのセクションは判断・行動につながる構造にする

---

## Phase 1: 準備

### 1-1. 次回会議日の決定
- 引数 `$ARGUMENTS` に `YYYYMMDD` が指定されていればその日付
- 未指定なら、今日から見て次の水曜日を計算

### 1-2. 前回の本部長会議ページを取得
- `mcp__notion__notion-fetch` で親ページ `29393c7ce32d80f4b140efccb98bf7a1` を取得
- 子ページ一覧から最新の `本部長会議_YYYYMMDD` を特定
- 特定できたらそのページも `mcp__notion__notion-fetch` で取得

### 1-3. 前回天気図の抽出
- 前回ページの「天気図サマリー」テーブルから、各本部の天気（☀/⛅/☁/☂）を抽出して保持
- 形式: `{ "カード事業本部": "☀", "支払い事業本部": "⛅", ... }`
- 前回ページがない場合は空（全本部 🆕 扱い）

### 1-4. 設定ファイル読み込み
- `~/.claude/private/weekly-ops-config.md` を Read で読み込み
- 本部→チャネルマッピング、DB ID、天気図ルールを取得

### 1-5. 対象期間の決定
- 前回会議日の翌日 〜 昨日（今日含まず）
- 前回ページが特定できない場合は、7日前〜昨日をデフォルト
- **対象期間の日数を計算し、Phase 3 サブエージェントに渡す**

## Phase 2: 前回アクション自動追跡

前回ページから情報を抽出:
1. 「前回アクション」テーブルの各行（アクション内容 + 担当者）
2. 会議中の「ネクストアクション」テーブルの各行（あれば）

各アクションについて、Slack で進捗を検索:
- `mcp__slack__slack_search_messages` で `from:@担当者名 キーワード after:前回会議日` を検索
- 検索結果からステータスを判定:
  - **完了**: 「完了」「done」「対応済」「リリース」等の言及あり → ✅完了
  - **進行中**: 言及はあるが完了示唆なし → 🔄進行中
  - **未着手**: 該当する言及が見つからない → ⬜未着手（ただし検索漏れの可能性あり）
  - **要確認**: 判定に自信がない → ❓要確認

**重要**: Slack検索で見つからない=未着手とは限らない。DM内の会話は検索にヒットしないことがある。判断に迷う場合は「❓要確認」とする。

結果をテーブル形式で保持:
```
| アクション | 担当 | 状況 | 根拠 |
```

```
## 出力ルール（厳守）

- 各本部のセクションは必ず番号付き（1〜7）で**1回ずつ**出力すること。同一セクションの重複出力は禁止
- 本部の出力順序は指定された順番を守ること
- Notionページ内では「支払い事業本部」と表記すること（「支払い.com」はURL自動リンクされるため禁止）

## 天気図判定ルール

- ☀: ポジティブトピックが大多数、インシデントなし
- ⛅: おおむね順調だが軽微な懸念あり
- ☁: 複数の課題・遅延あり、または情報不足
- ☂: 重大なインシデント・リスク・障害あり
- **対象期間が3日未満の場合**: チャネル投稿5件以上の本部を除き、天気図は一律 ☁（データ不足）とする
天気図の後に一言コメント（例: ☀ 売上好調）

各本部について以下を出力してください:

1. **天気図推定**（☀/⛅/☁/☂ + 一言コメント）

2. **3行サマリー**
   What: 今週何があったか（1行）
   Why: なぜそれが重要か（1行）
   So what: だから何をすべきか（1行）

3. **今週の前進**（P1-P3、優先度順）
   P1: 最も重要な前進
   P2: 次に重要
   P3: その次

4. **来週の課題**（P1-P3、優先度順）
   P1: ... Why: ...（具体的な期限があれば明記）
   P2: ... Why: ...
   P3: ... Why: ...

5. **議題・判断事項の候補**
   他本部や経営層に共有・判断を求めるべきトピックがあれば記載。各候補は以下の形式:
   - **議題**: 1行で何を決めるか
   - **背景**: なぜ今この判断が必要か（1行）
   - **選択肢**: A案 / B案（/ C案）— 各案の概要
   - **推奨**: どの案が望ましいか（根拠1行）
   - **関連本部**: どの本部が関わるか
   - **期限**: いつまでに決める必要があるか

6. **インシデント/リスク/クレーム候補**
   - リスク & インシデントDB に登録すべき事項
   - 労務インシデントに該当しそうな事項
   - 顧客クレームに該当しそうな事項

7. **判断リクエスト**（☂/☁ の場合のみ）
   この本部について、本部長会議で決めてほしいことを1〜2点、以下の形式で記載:
   - **問い**: 「〜すべきか？」「〜の方針をどうするか？」の形
   - **制約**: 期限・予算・リソース等の制約条件
   - **推奨案**: データから読み取れる推奨（あれば）
   ☀/⛅ の本部は「判断リクエストなし」と記載。

メッセージが少ない/無いチャネルは「（データ不足）」と記載。
```

## Phase 5: 週次ページ生成

収集したすべてのデータを統合し、新しい週次ページを作成する。

### 5-0. マージルール（重複防止）

- 各サブエージェントの出力をマージする際、同一本部の同一セクション（来週の課題等）が重複している場合は **最初の出現のみ採用** する
- 本部名の表記ゆれ（カード事業 / カード事業本部 等）は正式名称に統一

### 5-1. 期限アイテム抽出

各本部レポートの「来週の課題」「議題・判断事項の候補」と Notion DB のエントリから:
- 具体的な日付（期限）が記載されているアイテムを抽出
- 会議日から+7日以内の期限を持つものを対象
- 期限日の昇順でソート

### 5-2. アジェンダ自動プリセット（判断フレーム付き）

各エージェントの「議題・判断事項の候補」から:
- 期限が近いもの・複数本部に跨がるものを優先して最大3つ選出
- `【自動抽出】` プレフィックスを付けて手動追加と区別
- 各議題に **判断フレーム**（背景 / 選択肢 / 推奨 / 判断者）を付与
- 選出された議題は「議題・意思決定」テーブルにもプリセットする（結論列は空欄）

### 5-3. 天気図前回比の計算

Phase 1-3 で取得した前回天気図と今回の天気図を比較:
- 天気ランク: ☀=4, ⛅=3, ☁=2, ☂=1
- 推移: 今回 > 前回 → ↗（改善）、今回 = 前回 → →（維持）、今回 < 前回 → ↘（悪化）、前回データなし → 🆕

### 5-4. クロス本部イシュー抽出

各本部レポートの「議題・判断事項の候補」と「来週の課題」を横断的にスキャンし:
- **関連本部が2つ以上** のトピックを抽出
- 同一トピックが複数本部で言及されている場合も抽出
- 最大5件、重要度順でソート

### 5-5. ☂ トリアージ

☂ の本部が複数ある場合、以下の基準で「この会議で扱う」vs「個別フォロー」を判定:
- **この会議で扱う**: 他本部の協力が必要 / 経営判断が必要 / 期限が今週中
- **個別フォロー**: 当該本部内で対処可能 / 期限に余裕あり / 情報共有レベル
- 判定結果を天気図テーブルの一言コメントに `【要議論】` or `【個別FU】` で明記

### 5-6. コンテンツ構造

差分フォーカスの原則:
- ☂/☁ の本部 → 詳細展開（3行サマリー + 前進 + 課題 + 判断リクエストすべて表示）
- ☀/⛅ の本部 → 圧縮表示（What 1行 + 注目1点のみ。詳細はトグル内）

### 5-7. Notion Markdown

`mcp__notion__notion-create-pages` で新ページ作成:
- 親ページ: `29393c7ce32d80f4b140efccb98bf7a1`
- タイトル: `本部長会議_YYYYMMDD`（会議日付）
- アイコン: 🗓️
