## Step 2: データ統合 + DB更新

### 2a. スプシデータ + Legal管理シート + Metabase をorgIDでJOIN

各案件について以下を統合:
- 回収状況シート: orgID, 企業名, DPD, 債権額, 担当者, 先方主張, 支払予定日, 回収手段, ネクストアクションメモ, C列(回収状況)
- Legal管理シート: 案件ステータス_Legal, 保全ステータス, 本訴ステータス, 連帯保証
- Metabase（Step 1eで取得）: MT残高（orgIDでJOIN。該当なしはnull）

### 2b. フェーズ判定
DPDベースの簡易判定:
```
DPD 5-7  → フェーズ1（初期情報収集）
DPD 8-12 → フェーズ2（接触・主張確認）
DPD 13-19 → フェーズ3（裏付け確認・支払計画）
DPD 20-29 → フェーズ3 継続
DPD 30+  → フェーズ5（エスカレーション判定）
```
※ 回収手段 = 「分割合意」or「一括支払合意」→ フェーズ4に上書き（支払日が合意済み）
※ 債権額 >= 1,000万 → フェーズ5に上書き
※ 回収手段 = 「仮差押」「訴訟提起」「公正証書」はフェーズ上書きしない（法的措置はDPDベースのまま）

### 2c. Notion DB と照合

スプシデータの各案件 vs DB既存エントリを orgID × 回収期日 で照合:

**新規**（DBに未登録 かつ C列=未回収）:
→ `mcp__notion__notion-create-pages` で新規作成
→ ステータス = アクティブ
→ 担当者 = スプシの担当者名（テキスト）
→ 回収手段 = スプシの回収手段をselectにマッピング（case-config.md参照）。マッチしない場合はnull
→ サマリー = ネクストアクションメモがあればその内容、なければ `DPD{X} / {金額}万 / {回収手段orフェーズ} / {担当者}`
→ MT残高 = Metabaseから取得した値（該当orgIDがない場合はnull）
→ MT残高更新日 = 今日の日付（Metabase取得成功時のみ）
→ 保全実効性 = 下記「保全実効性の判定ロジック」で算出

**既存**（DBに登録済み）:
→ DPD, 債権額, 支払予定日, 担当者, 先方主張, 回収手段, 保全ステータス, 案件ステータス_Legal, 連帯保証, フェーズ, 最終更新日 を更新
→ サマリー = ネクストアクションメモがあればその内容、なければ `DPD{X} / {金額}万 / {回収手段orフェーズ} / {担当者}` で毎回更新
→ MT残高の更新（Metabase一括取得が成功した場合のみ。トークン失効時は全件スキップ）:
  - 該当orgIDがMetabase結果に存在 → MT残高 = 取得値, MT残高更新日 = 今日
  - 該当orgIDがMetabase結果に不在（MT未連携）→ MT残高 = null（既存値をクリアしない。そのまま保持）
→ 保全実効性 = 下記「保全実効性の判定ロジック」で算出（**新規・既存ともに毎回必ず再計算**）

### 保全実効性の判定ロジック（重要: 毎回全件に適用すること）
MT残高と債権額から以下の条件で判定し、selectプロパティに設定する:
```
if MT残高 == null:
    保全実効性 = "不明"     # MT連携なし or データ取得失敗
elif MT残高 >= 債権額 * 0.5:
    保全実効性 = "高"       # 仮差押で実効的な回収が見込める
elif MT残高 >= 1000000:     # 100万円以上
    保全実効性 = "中"       # 一定の回収が見込める
else:
    保全実効性 = "低"       # MT残高 < 100万。保全効果は限定的
```
**注意**: MT残高が数値として存在する場合（0を含む）は必ず「高」「中」「低」のいずれかになる。「不明」はMT残高=nullの場合のみ。
→ C列=回収済みの場合 → ステータスを「回収済み」に更新
→ フェーズが変わった場合は `phase_transitions` リストに追加

**回収済み判定**（DB既存エントリについて）:
→ 全行の回収状況マップ（フィルタ前）で orgID × 回収期日(±5日) を照合
→ C列=回収済み → ステータスを「回収済み」に更新、`recoveries` に追加
→ スプシのどのシートにもorgIDが完全に不在 → ステータスを「回収済み」に更新、`recoveries` に追加
→ C列=未回収だがフィルタ閾値（30万/DPD5）未達 → **変更なし**（既存ステータス維持）

### 2d. 初回スキャン時の特別処理
case-alert-last.md が存在しない場合:
- 全件を新規登録（上記2cの「新規」処理）
- シグナル検知はスキップ
- Slack通知は「初期セットアップ完了: XX件登録」の1通のみ
- case-alert-last.md を作成して終了

## Step 3: シグナル検知

前回スキャン結果（case-alert-last.md）と比較して**差分のみ**検知:

### シグナル1: DPD5+大口延滞（高）
- 条件: DPD >= 5 かつ 債権額 >= 10,000,000円
- 差分: 前回通知済みでないもの

### シグナル2: 支払予定日超過（高）
- 条件: 支払予定日 < 今日 かつ 支払予定日が非空
- 差分: 前回通知済みでないもの

### シグナル3: DPD30到達で未解決（高）
- 条件: DPD >= 30 かつ ステータス != 回収済み
- 差分: 前回通知済みでないもの

### シグナル4: 複数月延滞に新規移行（中）
- 条件: 複数月延滞シートの企業が前回スキャンのリストにない
- 差分: 前回リストと比較

### シグナル5: フェーズ遷移（通知）
- 条件: Step 2cで検知した `phase_transitions`
- フォーマット:
```
⚡ {企業名}: フェーズ{旧}→{新}に遷移（DPD{X}）
要対応: {フェーズに応じた1行要約}
詳細ガイド: /case-guide {企業名}
```

### シグナル6: 回収完了（ポジティブ通知）
- 条件: Step 2cで検知した `recoveries`

### シグナル7: 保全機会検知（高/中）
- 発火条件（全て AND）:
  - 前回MT残高 < 100万（or N/A） かつ 今回MT残高 >= 100万
  - 保全ステータス = なし or 空（保全未済）
  - DPD >= 13
  - 債権額 >= 100万
- 急げ度判定:
  - MT残高 > 債権額の50% → 🔴即日（Tier 1）
  - それ以外 → 🟡数日以内（Tier 2）
- 差分: 前回通知済みでないもの（通知済みシグナル「保全機会」リストで管理）
- フォーマット:
  `⚠️ 保全機会: {企業名}（MT残高 {X}万 / 債権額 {Y}万）→ 仮差押検討`

## Notes
- AskUserQuestion は使用しない（非対話で完結）
- データ取得時にエラーが出た場合はスキップし、取得できたデータのみで処理を継続
- 金額表示: 万円単位（小数点1桁）。例: 8,930,000 → 893.0万、30,000,000 → 3,000.0万
- DPDの計算: `(今日の日付 - 支払期日).days`。支払期日が解析できない場合はスキップ
- orgIDの数値比較: カンマを除去して整数化
- 初回スキャン時はシグナル検知をスキップし、DB初期登録のみ実行
- DB更新は1件ずつ順番に実行（Notion API負荷軽減）
- gogcli: `gog` コマンドがPATHに入っていること
- Task subagent は使用しない（直列実行で2-3分に収まる想定）
